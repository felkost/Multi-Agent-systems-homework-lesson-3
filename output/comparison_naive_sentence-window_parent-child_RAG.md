# Порівняння підходів naive, sentence-window та parent-child retrieval у RAG

## 1. Naive Retrieval

### Принцип роботи
- Прямий векторний пошук за запитом без додаткової обробки.
- Вибираються топ-K релевантних шматків (chunks) документів.
- Вибрані шматки передаються у генеративну модель для формування відповіді.

### Переваги
- Простота реалізації.
- Швидкість роботи.
- Підходить для простих, чітких запитів.

### Недоліки
- Погано працює з неоднозначними або складними запитами.
- Відсутність контексту для генерації відповіді.
- Слабка підтримка багатокрокового логічного виведення.

### Випадки використання
- Коли потрібна швидка відповідь на прості питання.
- Коли контекст не критичний для генерації.

---

## 2. Sentence-Window Retrieval

### Принцип роботи
- Індексація маленьких одиниць — речень (children).
- Пошук відбувається по реченнях.
- Для генерації відповіді передається більший контекст — вікно з кількох речень (parent).

### Переваги
- Висока точність пошуку на рівні речень.
- Більший контекст для генерації покращує якість відповіді.
- Баланс між точністю пошуку і контекстом генерації.

### Недоліки
- Потрібно налаштовувати розмір вікна для оптимального балансу.
- Може збільшувати обсяг даних для генерації, що впливає на продуктивність.

### Випадки використання
- Відкриті питання, де потрібен контекст для повної відповіді.
- Технічні пояснення, де важливий контекст навколо речення.

---

## 3. Parent-Child Retrieval

### Принцип роботи
- Дво-рівнева ієрархія: маленькі "дитячі" шматки індексуються для пошуку.
- При виборі "дитини" повертається більший "батьківський" шматок для генерації.
- Дедуплікація батьківських шматків, якщо кілька дітей одного батька потрапляють у топ.

### Переваги
- Поєднує точність пошуку маленьких шматків з багатством контексту великого батьківського шматка.
- Зменшує розмиття релевантності при пошуку по великих шматках.
- Гнучкість у налаштуванні розмірів child і parent.

### Недоліки
- Потрібна додаткова логіка для дедуплікації.
- Може збільшувати обсяг контексту для генерації, що впливає на продуктивність.

### Випадки використання
- Відкриті питання, сумаризація, технічні пояснення.
- Коли потрібен контекст для впевнених і повних відповідей.
- Не підходить для коротких відповідей або класифікації, де потрібен мінімальний контекст.

---

# Практичні рекомендації

- Починайте з naive RAG для швидкого прототипування і простих задач.
- Якщо відповіді недостатньо точні або повні, переходьте до sentence-window або parent-child retrieval.
- Використовуйте sentence-window, якщо важливо зберегти контекст на рівні речень і не перевантажувати генератор.
- Обирайте parent-child retrieval для складних завдань, де потрібен широкий контекст для генерації, наприклад, технічні пояснення або сумаризація.
- Налаштовуйте розмір "дитячих" і "батьківських" шматків, щоб балансувати між точністю пошуку і обсягом контексту.
- Впроваджуйте дедуплікацію батьківських шматків при parent-child retrieval, щоб уникнути надмірного дублювання контексту.
- Моніторте продуктивність і якість відповідей, щоб коригувати параметри chunking і retrieval.
- Уникайте parent-child retrieval для задач, де потрібна коротка відповідь або класифікація, щоб не перевантажувати модель зайвим контекстом.

---

# Висновок

- Naive RAG підходить для простих задач без потреби в контексті.
- Sentence-window і parent-child retrieval краще для складних, відкритих питань, де контекст важливий.
- Parent-child retrieval є більш гнучким і широко застосовується у виробничих системах.
- Вибір підходу залежить від задачі, бюджету на генерацію та необхідного контексту.
